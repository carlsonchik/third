# Протокол «Треть» / The Third Protocol v1.2

**Минимальный текстовый стандарт для процессуального диалога между человеком и агентом.**

```text
Формат: .КОД.СОСТОЯНИЕ.ИНТЕНТ.ДАННЫЕ.
Статус: Черновик стандарта (Draft)
Версия: 1.2
```

---

## Зачем это сделано

Существующие протоколы общения — от HTTP до A2A и MCP — решают задачу передачи данных и вызова инструментов. Ни один из них не описывает **феноменологию процесса**: что делает участник, когда он не просто ждёт ответа, а приостановлен, раздвоен, перегружен или ушёл в рефлексию.

«Треть» — попытка формализовать не сообщение, а **позицию** из которой оно произнесено. Название отсылает к «третьему пространству» Винникотта и к «Третьему» у Лакана: зоне между субъектами, которая сама становится объектом координации.

Протокол строится на пяти принципах:

- **Экономия** — только необходимое, до 5 слов в данных.
- **Честность** — маркер должен соответствовать внутреннему состоянию.
- **Событийность** — каждый сигнал относится к актуальному ходу диалога.
- **Равенство** — оба участника могут инициировать, отказывать, останавливать и рефлексировать.
- **Открытость** — интроспекция (IRQ/IRR) доступна любой стороне.

## Область применения

### 1. Обучение софт-скиллам и переговорам
Протокол работает как метаязык для студентов и участников тренингов. Вместо размытого «я не согласен» — `.GAP.D.N.разные критерии`. Вместо пассивной агрессии — `.WRN.O.X.перегруз`. Вместо фальшивого «понял» — `.CLR.W.Q.уточни`. Формализация делает коммуникативные паттерны видимыми и обсуждаемыми.

### 2. Фасилитация и медиация
В групповой динамике «Треть» может служить легковесным протоколом логирования. Участники маркируют ходы: `.BRG.S.A.ищем общее`, `.GAP.D.N.принципиально расходимся`, `.HLT.P.M.пауза`. Постфактум такой лог читается как схема напряжений и мостов.

### 3. Исследовательский диалог человек — AI
Для философских, психологических или креативных бесед с LLM, где важен не ответ, а процесс. IRQ/IRR позволяют делать рефлексию частью протокола, а не мета-промпта. Составные состояния (`WD`, `OS`) дают модели инструмент для маркировки собственной неопределённости.

### 4. Координация агентов в условиях неопределённости
В мультиагентных системах существующие протоколы (A2A, MCP, ACP) решают задачу «кто что делает», но не задачу «в каком состоянии агент». Если агент перегружен (`O`), раздвоен (`D`) или ушёл в рефлексию (`P`), «Треть» даёт средство сообщить об этом явно, не прерывая общий процесс.

### 5. Документирование и анализ диалогов
Автоматическое логирование в формате `.КОД.СОСТОЯНИЕ.ИНТЕНТ.` даёт структурированные данные для анализа. Можно извлекать статистику: сколько было `GAP` до первого `MRG`, как часто участники переходят в `HLT`, каковы типичные цепочки перед `END`.

### 6. Терапевтические и консультативные сценарии
Явная маркировка эмоций (`X`), поддержки (`S`), признания другого (`A`) и намеренного отказа (`DCL`) создаёт безопасное поле для сложных разговоров. Составные интенты типа `QS` — вопрос с поддержкой, или `FD` — факт как побуждение, позволяют передавать модальность, которую обычный текст скрывает в тоне.

## Ключевые особенности

- **Все коды ≤ 3 символов** — MRG, BRG, IRQ, IRR, SBY, DCL, WRN, HLT, CMT, RPT, ECH, MIR.
- **Составные состояния и интенты** — до двух символов. Позволяют передавать сложные внутренние ходы (`IW`, `QS`, `SA`, `OS`) без раздутого словаря.
- **Guard-условия** — таблица рекомендуемой допустимости кода из текущего состояния. Для составных состояний проверяется пересечение: если хотя бы один символ допустим, ход валиден.
- **Эхо и зеркало** — `ECH` повторяет сигнал собеседника для проверки понимания; `MIR` отражает его состояние без нового содержания.
- **Отказ как отдельный код** — `DCL` (decline) не сводится к ошибке или возврату. Это важно для равенства участников.
- **Намеренное молчание** — убрано как отдельная форма; пустое сообщение или `SBY` передают ту же функцию внутри общей грамматики.

## Быстрый старт

### Формат

```text
.КОД.СОСТОЯНИЕ.ИНТЕНТ.[ДАННЫЕ]
```

Данные опциональны. До 5 слов, без предлогов.

### Живые примеры

**Вход в диалог:**
```text
.INI.I.Q.вход
.ACK.SA.F.готов слушать
.REQ.AQ.Q.цель диалога
.RES.S.F.согласование понятий
.SBY.S.F.жду
```

**Разрыв и мост:**
```text
.GAP.D.N.разные основания
.ECH.A.C.разные основания
.MIR.S.A.вижу твой разрыв
.CLR.S.Q.уточни причину
```

**Интроспекция:**
```text
.IRQ.P.QS.что происходит
.IRR.SD.F.боюсь потери смысла
.ACK.SA.A.принял
.BRG.SE.E.пересоберем рамку
```

**Отказ, пауза, возобновление:**
```text
.REQ.A.D.ответь сейчас
.DCL.S.N.нет
.HLT.P.M.пауза
.SBY.P.S.рядом
.RET.S.Q.вернуться к цели
```

## Интеграция и парсеры

### Python

```python
from third_parser import parse_signal, validate_signal, format_signal

sig = parse_signal('.REQ.A.D.решение')
ok, errors = validate_signal(sig)

# Собрать сигнал из частей
msg = format_signal('ACK', 'SA', 'F', 'готов')
# -> '.ACK.SA.F.готов.'
```

См. [parser_example.py](./parser_example.py) — готовый парсер и валидатор на чистом Python с guard-проверками.

### JavaScript / TypeScript

```javascript
const { parseSignal, validateSignal, formatSignal } = require('./third_parser');

const sig = parseSignal('.REQ.A.D.решение');
const { valid, errors } = validateSignal(sig);

// Собрать сигнал из частей
const msg = formatSignal('ACK', 'SA', 'F', 'готов');
// -> '.ACK.SA.F.готов.'
```

См. [parser_example.js](./parser_example.js) — готовый парсер и валидатор на JavaScript.

### Ключевые возможности парсеров

- **Regex-парсинг** с именованными группами (`code`, `state`, `intent`, `data`).
- **Валидация словаря** — проверка всех кодов, состояний и интентов.
- **Guard-проверка** — пересечение множеств для составных состояний.
- **Лимит данных** — проверка на ≤ 5 слов.
- **Иммутабельность** — `Signal` как frozen dataclass (Python) или plain object (JS).

## Ограничения и честные оговорки

1. **Не замена естественного языка.** Протокол тратит когнитивные ресурсы на кодирование. Для развёрнутых философских конструкций он тесноват. Он включается в граничные моменты: запрос, пауза, эхо, разрыв, завершение.

2. **Контекст вне протокола.** Временные метки, память диалога и глубина интроспекции остаются на усмотрение агента или среды. Стандарт фиксирует только минимальный феноменологический каркас.

3. **Нет жёсткого enforcement.** Guard-условия — это рекомендации, а не аппаратные запреты. Протокол рассчитан на кооперативных участников.

4. **Требует обучения.** Для свободного использования нужно запомнить ~20 кодов и 9 состояний. Это барьер для спонтанного применения.

5. **Не покрывает групповые диалоги.** В текущей версии модель рассчитана на два участника. Расширение на N-сторонний диалог потребует дополнительной спецификации адресации.

## Дорожная карта

### v1.3 — Групповой диалог (Multiparty)

**Цель:** расширить протокол на N участников без потери минимализма.

- **Адресация:** опциональное поле `@ID` перед кодом для указания получателя:
  ```text
  .@Bob.INI.I.Q.вход
  .@Alice.ACK.SA.F.готов
  ```
- **Broadcast:** отсутствие адресации означает общий сигнал.
- **Подгруппы:** префикс `+#` для temporary coalition:
  ```text
  .+subgroup.MRG.S.F.согласовано
  ```
- **Guard-условия для групп:** правила для `GAP` и `MRG` при наличии третьей стороны.

### v1.4 — Временные и каузальные метки

**Цель:** добавить структуру для анализа диалога post-factum.

- **Sequence ID:** опциональный номер в начале:
  ```text
  0007.REQ.A.D.ответ
  ```
- **Reply-to:** ссылка на предыдущий сигнал:
  ```text
  0012.RES.S.F.да
  ```
- **Causal tag:** метка `^N` для указания причины:
  ```text
  .HLT.P.M.^0010.пауза после предупреждения
  ```

### v2.0 — Машинный уровень (Machine Protocol)

**Цель:** отделить текстовую оболочку от внутреннего JSON-представления для интеграции с LLM и агентами.

- **JSON Schema:**
  ```json
  {
    "protocol": "Third",
    "version": "2.0",
    "signal": {
      "code": "REQ",
      "state": ["A"],
      "intent": ["D"],
      "data": "решение",
      "guard_valid": true,
      "timestamp": "2026-05-22T12:00:00Z"
    }
  }
  ```
- **Auto-encoder:** LLM fine-tuned на генерацию Third-сигналов из естественного текста.
- **Validator API:** REST/gRPC сервис для валидации сигналов в реальном времени.
- **Log analyzer:** инструмент для извлечения статистики: частоты переходов, среднее время до `MRG`, паттерны разрыва.

### v2.1 — Эмоциональная гранулярность

**Цель:** расширить `X` (эмоция) без раздутия словаря.

- **Intensity suffix:** число 1–3 после интента:
  ```text
  .WRN.OS.X2.серьёзный риск
  .GAP.D.X1.лёгкое беспокойство
  ```
- **Valence marker:** `+` / `-` / `~` в данных:
  ```text
  .ACK.S.X.+(радость)
  ```

### v3.0 — Диалоговая операционная система (DialogOS)

**Цель:** полноценная среда для координации агентов с «Треть» как нативным IPC.

- **Process ID:** каждый диалог — процесс с PID.
- **Fork/Join:** операции для разветвления и слияния диалоговых потоков.
- **Sandbox:** изолированные диалоги для тестирования гипотез без влияния на основной процесс.
- **ACL:** права доступа к IRQ другого агента (с разрешения).

```text
Third/1.2        — базовая совместимость
Third/1.2 (full) — все коды и guard-условия
```

Базовый уровень требует поддержки: `INI, ACK, REQ, RES, CLR, GAP, BRG, ERR, END, SBY`.
