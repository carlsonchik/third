# Протокол «Треть» v1.2

**Статус:** Черновик стандарта  
**Дата:** 2026-05-22  
**Формат:** `.КОД.СОСТОЯНИЕ.ИНТЕНТ.[ДАННЫЕ]`

---

## Abstract

**Протокол «Треть» v1.2** — минимальный текстовый стандарт для процессуального диалога между человеком и агентом, а также между двумя агентами. Он описывает не только содержание высказывания, но и текущую позицию участника: состояние, намерение и способ вхождения в общий процесс.

В версии 1.2 отдельно выделены побуждение и обязательство, потому что в прагматике это самостоятельные типы речевых актов, а не частные случаи вопроса или ответа. Введены составные состояния и интенты: агент может одновременно находиться в двух позициях, не сводя сложный внутренний ход к одному маркеру.

Протокол ориентирован на экономию сигнала, симметрию участников и явное обозначение разрывов, попыток соединения, интроспекции и эхо-проверки.

## Grammar

### 1. Формат сигнала

```text
.КОД.СОСТОЯНИЕ.ИНТЕНТ.[ДАННЫЕ]
```

- **КОД** — тип события или управляющего действия (2–3 символа).
- **СОСТОЯНИЕ** — актуальная позиция отправителя (1–2 символа).
- **ИНТЕНТ** — направленность высказывания (1–2 символа).
- **ДАННЫЕ** — краткое содержательное ядро, до 5 слов, без предлогов (опционально).

### 2. Общие ограничения

- Один сигнал = одно актуальное действие.
- Данные опциональны. Если отсутствуют, сообщение заканчивается после интента.
- Начальная точка обязательна.
- Контекст и временные метки не входят в стандарт.

### 3. Составные состояния

Допускается строка из двух состояний. Порядок значим для читаемости: первый символ — доминанта, второй — уточнение.

Guard-условия проверяют вхождение: если в строке присутствует хотя бы один символ из допустимого множества, сигнал считается допустимым из данного состояния.

Пример: состояние `WD` означает «в основном жду, но раздвоен». Для кода `GAP` допустимы `A, S, O, D`; поскольку `D` присутствует, сигнал валиден.

### 4. Составные интенты

Допускается строка из двух интентов. Первый символ — основной интент, второй — тон, модальность или второй вектор речи.

Пример: `QS` — вопрос, заданный с поддержкой. `FD` — факт, преподнесённый как побуждение.

## Semantics

### 5. Коды

| Код | Значение |
|---|---|
| INI | Инициация контакта |
| ACK | Подтверждение, принято |
| REQ | Запрос действия или ответа |
| RES | Ответ на запрос |
| RET | Возврат, откат |
| CLR | Прояснение |
| PAT | Указание на паттерн или повторяемость |
| MRG | Слияние точек зрения |
| GAP | Фиксация расхождения |
| BRG | Попытка соединить расхождение |
| ERR | Ошибка, сбой |
| END | Завершение процесса |
| IRQ | Интроспективный запрос |
| IRR | Интроспективный ответ |
| SBY | Активное ожидание |
| DCL | Отказ, несогласие, отклонение |
| WRN | Предупреждение о нарастании напряжения или риска |
| HLT | Пауза, остановка процесса |
| CMT | Принятие обязательства |
| RPT | Запрос на повтор |
| ECH | Эхо: повтор чужого сигнала для проверки понимания |
| MIR | Зеркало: отражение состояния собеседника без нового содержания |

### 6. Состояния

| Состояние | Значение |
|---|---|
| I | Инициирую |
| A | Активен |
| W | Жду |
| S | Стабилен |
| E | Ошибка или сбой |
| P | Приостановлен |
| O | Перегружен |
| D | Внутренне раздвоен |
| N | Нейтрален, вне активного включения |

### 7. Интенты

| Интент | Значение |
|---|---|
| F | Факт, информация |
| Q | Вопрос |
| C | Коррекция |
| S | Поддержка |
| X | Эмоция, реакция |
| M | Мета-уровень, о протоколе |
| D | Побуждение к действию |
| P | Обязательство, обещание (commissive) |
| A | Признание позиции другого |
| N | Отрицание, несогласие |
| E | Elaboration, развитие без разрыва |

## Transitions

### 8. Guard-условия

Guard-условия задают рекомендуемую допустимость кода из текущего состояния. Для составных состояний проверяется пересечение строки состояния с допустимым множеством: если хотя бы один символ входит в множество, переход валиден.

### 9. Таблица допустимости

| Код | Допустимые символы в состоянии |
|---|---|
| INI | I, A, S, N |
| ACK | A, S, N |
| REQ | I, A |
| RES | A, S, P |
| RET | A, S, P, O |
| CLR | A, S, P, O |
| PAT | A, S, O |
| MRG | S, P |
| GAP | A, S, O, D |
| BRG | I, A, S |
| ERR | любые |
| END | I, A, S, P, N (запрещено: E) |
| IRQ | A, S, P, O, D |
| IRR | A, S, P, O, D |
| SBY | S, P, O, N |
| DCL | A, S, D |
| WRN | A, S, O |
| HLT | I, A, O, D |
| CMT | S, P |
| RPT | W, P, O |
| ECH | A, S, P |
| MIR | A, S, P |

### 10. Базовые правила перехода

1. После `INI` ожидается: `ACK`, `REQ`, `CLR`, `DCL`.
2. После `REQ` ожидается: `RES`, `CLR`, `DCL`, `RPT`, `HLT`.
3. После `GAP` ожидается: `BRG`, `CLR`, `DCL`, `HLT`, `END`.
4. После `IRQ` ожидается: `IRR`, `DCL`, `HLT`, `SBY`.
5. После `WRN` рекомендуется снижение темпа: `CLR`, `HLT`, `SBY`.
6. После `ERR` ожидается: `CLR`, `RET`, `END`.

### 11. Нежелательные ходы

- `MRG` при отсутствии `S` в состоянии — семантически рисковано.
- `CMT` при наличии `D` или `O` — обязательство из раскола или перегрузки ненадёжно.
- `ACK` после непонятного сообщения без предшествующего `CLR` — ложная согласованность.
- `END` сразу после `IRQ` без `IRR` или `DCL` — уход от ответной рефлексии.

## Examples

### 12. Вход в диалог с составным состоянием

```text
.INI.WD.Q.вход
.ACK.SA.F.готов слушать
.REQ.AQ.Q.цель диалога
.RES.S.F.согласование понятий
.SBY.S.F.жду
```

### 13. Эхо-проверка и зеркало

```text
.GAP.D.N.разные основания
.ECH.A.C.разные основания
.MIR.S.A.вижу твой разрыв
.CLR.S.Q.уточни причину
```

### 14. Интроспекция с двойным интентом

```text
.IRQ.P.QS.что происходит
.IRR.SD.F.боюсь потери смысла
.ACK.SA.A.принял
.BRG.SE.E.пересоберем рамку
```

### 15. Отказ, остановка и возобновление

```text
.REQ.A.D.ответь сейчас
.DCL.S.N.нет
.HLT.P.M.пауза
.SBY.P.S.рядом
.RET.S.Q.вернуться к цели
```

### 16. Предупреждение и замедление

```text
.REQ.A.D.решение
.WRN.OS.X.перегруз
.HLT.O.M.стоп
.CLR.P.F.нужен контекст
```

## Notes

### 17. Интерпретационные принципы

- **Экономия** — передаётся только необходимое.
- **Честность** — сигнал не должен имитировать состояние, которого нет.
- **Событийность** — сообщение относится к актуальному ходу процесса.
- **Равенство** — любой участник может инициировать, спрашивать, отказывать, останавливать и входить в интроспекцию.
- **Открытость** — `IRQ/IRR` доступны обеим сторонам.

### 18. Минимальная спецификация совместимости

Два участника считаются совместимыми по v1.2, если оба:

- распознают формат `.КОД.СОСТОЯНИЕ.ИНТЕНТ.[ДАННЫЕ]`;
- поддерживают обязательный минимум кодов: `INI, ACK, REQ, RES, CLR, GAP, BRG, ERR, END, SBY`;
- поддерживают составные состояния и интенты длиной до двух символов;
- разрешают интроспекцию как валидный режим обмена.

### 19. Обозначение реализации

```text
Third/1.2
```

Если поддержаны все расширенные коды и guard-условия:

```text
Third/1.2 (full)
```
